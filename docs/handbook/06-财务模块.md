# 财务模块分析（对账/付款/发票/结算）

## 模块组成

- **Reconciliation** - 对账管理
- **Payment** - 付款申请
- **Invoice** - 出口发票（收入侧留痕）
- **Settlement** - 结算清单（成本/毛利/净利分析）

---

## 一、对账（Reconciliation）

### 对账类型体系

| 类型 | 代码 | 说明 |
|------|------|------|
| 合同对账 | CONTRACT | 有合同的货款对账，以出货批次快照金额求和 |
| 无合同对账 | NO_CONTRACT | 无合同的费用对账，以费用明细行求和 |
| 工时对账 | LABOR | 样衣打样工时对账，版师多款合并 |

### 核心业务规则

| 规则 | 代码位置 |
|------|----------|
| 发票号全局查重，防同一发票重复对账/重复付款 | reconciliation.service.ts:32-38 |
| 一张对账单不可混含材料+加工两类合同 | reconciliation.service.ts:60-69 |
| 补料合同可并入母合同（核算口径归母） | reconciliation.service.ts:42-49 |
| 工时对账须同一版师、样衣均已完成对账 | reconciliation.service.ts:161-178 |
| 退回后释放发货批次，防同批次两单重复计费 | reconciliation.service.ts:329, 335-344 |

### 对账单状态机

```
DRAFT --[业务员submit]--> PENDING --[主管confirm]--> CONFIRMED --[付款完成]--> PAID
  ^                         |
  +------[主管reject]-------+  （退回至DRAFT，释放批次）
```

### 对账确认后的联动

1. 同订单已确认结算单标 `needs_recalc=1`
2. 合同门户状态推进至 `RECONCILED`
3. 订单状态从 `PRODUCING` → `DONE`

### 超发闸门

累计实发超过合同量+0.01时，确认时须填写 `over_reason`（超发放行原因留痕）

## 二、付款申请（Payment）

### 付款申请状态机

```
DRAFT --[submit]--> PENDING --[approve]--> APPROVED --[markPaid/addPaymentRecord]--> PAID
                       |
                       +--[reject]--> REJECTED
```

### 审批流

| 操作 | 前置状态 | 目标状态 | 角色 |
|------|----------|----------|------|
| 创建 | - | DRAFT | ADMIN/FINANCE/BUSINESS |
| 提交 | DRAFT | PENDING | ADMIN/FINANCE |
| 审批通过 | PENDING | APPROVED | ADMIN/FINANCE |
| 驳回 | PENDING | REJECTED | ADMIN/FINANCE |
| 标记已付 | APPROVED | PAID | ADMIN/FINANCE |

### 超付拦截机制（三层防护）

1. **创建时累计校验**（悲观锁串行化）
   - 锁定对账单行 `pessimistic_write`
   - `已有申请合计 + 本次金额 > 对账总额 + 0.01` 则拦截
   - 位置：payment.service.ts:110-125

2. **预付款冲抵校验**
   - `prepay_offset` 不能超过付款申请金额
   - `prepay_offset` 不能超过该工厂可用预付款余额
   - 位置：payment.service.ts:86-98

3. **分批付款累计校验**（悲观锁串行化）
   - `paid_total + 本次金额 > 实际应付额 + 0.01` 则拦截
   - 位置：payment.service.ts:166-184

### 付款完成后联动

- 付款申请付清（PAID）时，关联对账单从 `CONFIRMED` → `PAID`
- 付款完成后，同订单已确认结算单标 `needs_recalc=1`

### 预付款冲抵机制

- 审批通过时自动按 FIFO 顺序扣减预付款余额
- 悲观锁串行化，防止并发冲抵超扣

## 三、发票校验逻辑

### 发票金额校验（发票 = 对账金额 ± 0.01）

```ts
if (Math.abs(+(rec.invoice_diff ?? 0)) > 0.01) {
  throw new BadRequestException(`发票金额 ${rec.invoice_amount} 须与对账金额 ${rec.total_amount} 一致（允许±0.01）`);
}
```

### 发票差额计算

- 创建对账单时自动计算：`invoice_diff = invoice_amount - total_amount`
- `has_invoice` 标记：有发票号=1，无=0

### 出口发票（独立于对账发票）

- 出口发票（ExportInvoice）是收入侧留痕，支持"一票多款"
- 发票号唯一性校验
- 收汇登记后不可删除发票
- 收汇按款项行金额占比分摊到订单（拼柜场景）

## 四、结算清单（Settlement）

### 总货款聚合

- 按订单聚合同合同的已付款(`PAID`)对账单
- 已确认未付 (`CONFIRMED`) 单独返回，不计入总货款

### 核心计算公式

| 指标 | 公式 | 代码位置 |
|------|------|----------|
| 成本单价(含税) | `总货款含税 / 出货件数` | settlement.service.ts:72 |
| 成本单价(不含税) | `总货款不含税 / 出货件数` | settlement.service.ts:73 |
| 美金单价 | `发票金额USD / 出货件数` | settlement.service.ts:74 |
| 毛利 | `结算金额 - 总货款不含税` | settlement.service.ts:75 |
| 毛利率% | `毛利 / 结算金额 × 100` | settlement.service.ts:76 |
| 财务及管理费 | `结算金额 × 费率%(默认7%)` | settlement.service.ts:77 |
| 净利(不含退税) | `毛利 - 期间费用合计 - 财务管理费` | settlement.service.ts:78 |
| 净利(含退税) | `净利(不含退税) + 出口退税` | settlement.service.ts:79 |
| 保本汇率(含税) | `成本单价含税 / 美金单价` | settlement.service.ts:80-81 |
| 退税额 | `可退税不含税采购额 × 退税率` | settlement.service.ts:336 |

### 税额换算规则

| 情形 | 不含税金额计算 |
|------|---------------|
| 有票 (has_invoice=1) | `amount / (1 + tax_rate / 100)` |
| 无票 (has_invoice=0) | `amount`（按含税全额计入，防毛利虚高） |

### 期间费用四分桶

| 桶 | 关键词 |
|----|--------|
| 运杂(freight) | 运/货代/物流/拖车/报关 |
| 快邮(express) | 快递/快邮/邮寄/邮费 |
| 打样(sample) | 打样/样品/样衣 |
| 其它(other) | 以上均不匹配 |

### 同款多订单分摊

期间费用按出货份额分摊：`share = 本订单出货件数 / 全款总出货件数`，无出货则均分

### 结算单状态机

```
DRAFT --[confirm]--> CONFIRMED --[reopen]--> DRAFT（红冲重开，版本快照留痕）
```

## 五、超付拦截机制汇总

| 防护层 | 机制 | 代码位置 |
|--------|------|----------|
| 对账单创建 | 发票号全局查重 | reconciliation.service.ts:33-38 |
| 对账单确认 | 发票金额校验 ±0.01 | reconciliation.service.ts:378-382 |
| 付款申请创建 | 悲观锁 + 累计申请额不超过对账总额 | payment.service.ts:110-125 |
| 分批付款登记 | 悲观锁 + 累计付款不超过应付额 | payment.service.ts:166-184 |
| 预付款冲抵 | 不超过工厂可用余额 + FIFO扣减 | payment.service.ts:91-98, 359-381 |
| 工时对账 | 防重复：样衣不得已在其他工时对账单 | reconciliation.service.ts:179-187 |
| 退回防重复 | 退回后释放批次，再提交时校验批次仍被持有 | reconciliation.service.ts:335-344 |
| 结算软锁 | 对账/付款变动后标 needs_recalc=1 | reconciliation.service.ts:387-393 |

## 六、模块间依赖关系

```
┌─────────────┐
│  Order      │
│  (订单)     │
└──────┬──────┘
       │ order_id
       ▼
┌─────────────┐     contract_id      ┌──────────────┐
│  Reconcile  │ ◄─────────────────── │  Contract    │
│  (对账)     │                      │  (合同)      │
└──┬──────┬───┘                      └──────────────┘
   │      │
   │      │ reconcile_id
   │      ▼
   │   ┌─────────────┐
   │   │  Payment    │
   │   │  (付款)     │
   │   └─────────────┘
   │
   │ reconcile_id / contract_id
   ▼
┌─────────────┐    imports      ┌──────────────┐
│  Settlement │ ◄────────────── │ ExportInvoice│
│  (结算)     │   allocated-    │ (出口发票)   │
│             │   Receipts()    └──────────────┘
└─────────────┘
```

### 跨模块状态联动

1. 对账确认 → 合同门户 `RECONCILED` + 订单 `DONE` + 结算 `needs_recalc=1`
2. 付款完成 → 对账 `PAID` + 结算 `needs_recalc=1`
3. 结算 `needs_recalc=1` → 刷新付款汇总时清除
