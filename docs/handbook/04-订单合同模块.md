# 订单管理 & 合同管理模块分析

## 模块组成

- **Order** - 订单管理（5个状态的状态机）
- **Contract** - 合同管理（材料/加工/补料三种类型）

---

## 一、订单状态机

### 状态枚举

```
DRAFT(草稿) → CONFIRMED(已下单) → CONTRACTED(已生成合同) → PRODUCING(生产中) → DONE(已完成)
```

### 状态转移表

```ts
const STATUS_TRANSITIONS: Record<OrderStatus, OrderStatus> = {
  DRAFT → CONFIRMED,
  CONFIRMED → CONTRACTED,
  CONTRACTED → PRODUCING,
  PRODUCING → DONE,
  DONE → DONE,  // 终态自循环
};
```

### 关键设计要点

1. **唯一手动推进点**：只有 `DRAFT → CONFIRMED`（下单）是用户主动操作
2. **下游事件自动驱动**：
   - `CONFIRMED → CONTRACTED`：合同创建时自动回写（仅非补料合同触发）
   - `CONTRACTED → PRODUCING`：首次添加出货记录时自动推进
   - `PRODUCING → DONE`：由下游对账/结算事件完成闭环
3. **金额阈值审批**：草稿下单时，若 `total_amount > approval.order.threshold` 或佣金率超阈值 → 自动置 `approval_status=PENDING`

## 二、合同类型及区别

| 类型 | 枚举值 | 含义 | 单号规则 | 默认账期 | 交货偏移 |
|------|--------|------|----------|----------|----------|
| 材料合同 | MATERIAL | 面料/辅料购销 | HT-日期-序号 | 90天 | 订单交期 -45天 |
| 加工合同 | PROCESS | 委外加工 | HT-日期-序号 | 45天 | 订单交期 -10天 |
| 补料合同 | SUPPLEMENT | 材料补充 | 补料-母合同号-序号 | 90天 | 订单交期 -45天 |

### 关键差异

1. **生成方式**：
   - 材料合同：由订单按供应商拆单自动生成
   - 加工合同：手工创建（跳转新页面带入订单信息）
   - 补料合同：从已有材料合同复制（`parent_id` 指向母合同）

2. **数量来源**：
   - 材料合同数量 = 订单采购量（含损耗）
   - 加工合同数量 = 订单大货数

3. **甲乙方**：
   - 材料合同：甲方=供应商，乙方=本司
   - 加工合同：受托方=加工厂，委托方=本司

4. **价格包含项**：仅加工合同有 `price_includes`（工缴/裁剪/线/后道/运费等，默认10项）

5. **补料合同不改订单状态**：仅首次材料/加工合同生成时推进订单到 CONTRACTED

## 三、付款条款计算逻辑

### 字段定义

```ts
deposit_ratio (定金%)  默认 30
mid_ratio    (中期%)  默认 40
final_ratio  (尾款%)  默认 30
```

### 校验规则

```ts
if (Math.abs(deposit_ratio + mid_ratio + final_ratio - 100) > 0.01) {
  throw new BadRequestException('定金比例 + 中期比例 + 尾款比例必须等于 100%');
}
```

### 付款金额计算

- 定金 = `total_amount × deposit_ratio / 100`
- 中期款 = `total_amount × mid_ratio / 100`
- 尾款 = `total_amount × final_ratio / 100`

## 四、账期设置

### 默认账期

```ts
const DEFAULT_ACCOUNT_PERIOD_DAYS: Record<ContractType, number> = {
  MATERIAL: 90,     // 材料：发货日 + 90天
  PROCESS: 45,      // 加工：发货日 + 45天
  SUPPLEMENT: 90,   // 补料：发货日 + 90天
};
```

**到期日计算**：`due_date = 最后发货日(last_ship_date) + account_period_days`

## 五、从报价导入订单的数据流

### 入口

`PATCH /orders/:id/import-quote/:quoteId`

### 数据映射

```
报价(Quotation) → 订单(Order)  +  报价明细(QuotationItem) → 订单材料(OrderMaterial)
```

### 具体步骤

1. **前置校验**：订单必须为 DRAFT 状态，报价必须为 QUOTED/ADJUSTING/ORDERED 状态

2. **基础字段快照**：
   - `customer_id`, `middleman_name`, `buyer_id`, `buyer_name` ← 报价直接带入
   - `currency` ← 强制设为 `CNY`
   - `unit_price` ← 取 `quote.rmb_total`
   - `quote_synced_at` ← 记录导入时间

3. **材料明细重建**：
   - 清空订单现有材料行
   - 遍历报价明细，映射为订单材料
   - 调用 `buildMaterials()` 计算采购量

## 六、订单→合同拆单逻辑

### 材料合同生成

```
订单材料 → 按供应商分组 → 为每个供应商生成一份材料合同
  - 数量 = 订单采购量（含损耗）
  - 单价 = 订单材料单价
  - 分色/分码材料行展开
```

### 加工合同生成

```
手工创建 → 带入订单信息
  - 数量 = 订单大货数
  - 单价 = 手工输入
```

## 八、模块间依赖关系

```
Quote(报价) ──importFromQuote──→ Order(订单) ──generateFromOrder──→ Contract(合同)
                                                          │
                                          ┌───────────────┼───────────────┐
                                          ↓               ↓               ↓
                                   ContractMaterial  ContractShipment  ContractPortalLog
                                          │               │
                                          ↓               ↓
                                   Reconciliation(对账)  Settlement(结算)
```

### 数据流总结

- 报价 → 订单: 基础字段快照 + 材料明细复制
- 订单 → 合同: 按供应商分组拆单（材料）/ 手工创建（加工）
- 合同 → 门户: 推送→盖章→发货→对账→付款全流程闭环
- 合同→订单反写: 首次非补料合同生成时，订单状态 `CONFIRMED → CONTRACTED`
- 订单→报价反写: 下单时报价状态 `QUOTED → ORDERED`
