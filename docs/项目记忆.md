## 一、项目概览

面向服装外贸/制造企业的全流程数字化协同系统，覆盖 **样衣 → 报价 → 订单 → 合同 → 供应商门户 → 对账 → 付款 → 结算** 的完整闭环。目标：**字段级 / 公式级 / 状态机级对齐业务设计稿的产品级实现**。

- 8 张核心单据 + 基础资料（工厂/客户）+ 本司主体 + 报表 KPI。
- 两个前端：PC 管理端 `web`（Vue3 + Element Plus）、供应商 H5 门户 `portal`（Vue3 + Vant）。
- 后端 15 个模块、约 31 张实体表。

## 二、技术架构

- **monorepo（pnpm）**：`packages/{types,api,web,portal}`；`types` 改动后必须先构建。
- **后端**：NestJS 10 + TypeORM + MySQL 8 + Redis 7；全局前缀 `api/v1`；全局 `ValidationPipe`（`whitelist + forbidNonWhitelisted`）+ `ResponseInterceptor`。
- **鉴权**：JWT，载荷含 `role` + `type`（`admin` | `supplier`）；`@Roles` + `RolesGuard`；供应商 token 自动挡在管理端接口外。
- **部署**：单机 All-in-One（单台 ECS）。nginx 反代 `/api`→:3000，web/portal 静态；MySQL/Redis 走 docker-compose（仅绑 127.0.0.1）；API 走 systemd `i9-api`。运维脚本全在 `infra/scripts/`。

## 三、已交付模块与关键业务规则

### 基础资料
- **工厂**：主身份 + 附加身份（**双身份**，下拉按主/附加命中）；联系人子表；门户账号建档开通；编号 `S`（全局递增）。DTO 用 camelCase（`shortName/extraTypes/canInvoice`）。
- **客户**：按类型前缀 **中间商 `CM-` / 最终买家 `FE-`**；银行/联系人/快递地址子表。
- **本司主体**：多主体 + 默认主体，供 PDF 抬头 / 合同甲方 / 开票抬头取数。

### ① 样衣
- 7 态状态机；材料明细子表；寄样跟踪；**业务 / 版师双视图**（版师视图对客成本脱敏）。编号 `S-日期-序号`。

### ② 客户报价
- 18 字段；**报价明细 + 费用明细双子表**；**含损金额公式**；从样衣导入；状态机；下单反写「已成单」。对客合计对版师/打样脱敏。编号 `Q`。

### ③ 订单
- **采购量 = 用量 ×(1+损耗率)，整数进 1**；尺码搭配矩阵 + 材料明细；材料供应商从工厂库点选（配合供应商拆单）；从报价导入；佣金 / 生产工厂（生产工厂选择器只过滤委外加工商）。
- **状态机后两态由下游事件回写**：发货驱动「生产中」、对账驱动「已完成」。编号 `O`。

### ④ 合同
- 三类：材料 / 加工 / 补料。材料合同**自动从订单用料核算带出快照**（打通报价→合同链路）；加工合同数量取大货数；数量来源标注。
- 尾款比例（定金/中期/尾款）；**账期 材料 90 / 加工 45 天**；供应商拆单生成材料合同（按订单材料供应商分组）。
- **补料合同独立编号 `补料-母合同号-序号`**（不占 HT 主序号）；主合同编号 `HT-日期-序号`。
- 推送前校验工厂已绑定门户账号（**A5 死流程防护**）；**金额阈值审批**；**撤销推送/改后重推**（`revised` 标记，门户提示「合同已更新」，盖章后清零）。

### ⑤ 供应商门户（H5）
- **四步顺序锁定：盖章 → 发货 → 对账 → 开票**（靛蓝品牌色，对齐设计稿 v2.2）。
- 盖章：锁材料明细快照 + **须勾选「已阅读并同意合同条款」**；加工厂可见订单明细同步。
- 发货：**批次累计发货 + 超合同量确认 + 逐批锁价（快照当时单价）+ 收货地址**；回写最后发货日与到期日。
- 开票须**对账后**方可；发票附件上传（H5 van-uploader → `/uploads` → `invoice_url`）。

### ⑥ 对账
- 三类：合同 / 无合同 / 工时。**发票金额 = 对账金额 ±0.01 强校验**；**发票号全局查重**（防重复对账/付款）。
- **二级审批**：业务员初审提交 → 主管复核确认；**整单退回 + 逐批批注**；一单多合同（批次各带来源合同/款号，款号汇总）；款号检索；费用明细 / 工时明细子表；工时对账多款合并。
- 编号 `DZ-款号-序号`；发货单 `FH-款号-序号`。对账确认推进合同门户至「已对账」。

### ⑦ 付款
- 预付款 + 付款申请；审批流（草稿→待审批→已批准/驳回→已付款）。
- **预付款冲抵**（超可用余额拦截）；**分批付款超付拦截**（累计申请额 ≤ 对账应付）；新建时**提示存在预付余额 + 一键冲抵**；**工厂 + 申请日期组合检索**；**水单支持点击/拖拽/Ctrl+V 粘贴上传**。付款完成联动对账单「已付款」。编号 `PR`（无合同 `PR-NC`）。

### ⑧ 结算
- **财务口径**：财务费 7% / 结算金额 = 收汇 × 汇率 / 保本汇率 / 无票计税 / **退税自动测算**（可退税不含税采购额 × 退税率）。
- 成本明细**从对账付款按款号自动汇总**；**各行税率**（有票按此换不含税）；**按客户维度利润汇总**；两步态。编号 `JS-款号-序号`（rec：便于按款检索）。

### 报表 / KPI
- 转化漏斗 + 成单率 + 利润汇总（**亏损预警**）；真实数据工作台仪表盘。

### 产品级平台能力
- 文件上传（图片/PDF/Excel ≤20MB，`/uploads` 端点 + 可复用组件）；工厂/客户 **CSV 批量导入向导**（模板→上传/粘贴→校验→入库，逐行结果，零新依赖）；供应商门户发票上传闭环。

## 四、关键横切规则（改动前务必对齐）

- **编号（`NumberingService`）**：`next(prefix)`=`前缀-日期-序号`；`nextGlobal`=全局递增；`nextWithSegment(prefix,段)`=`前缀-段-序号`（段=款号等，**已限长 20** 防溢出）。前缀见 `NUM_PREFIX`。
- **RBAC**：6 内部角色 `ADMIN/BUSINESS/FINANCE/PATTERNMAKER/SUPERVISOR/SAMPLE_MAKER` + 供应商门户账号；`type=admin|supplier` 双体系。
- **响应封装**：`ResponseInterceptor` 统一 `{code,msg,data}`；**分页 `{items,total,page,size}` 会被展开为顶层 `total/page/size` + `data=items`**——断言/取数用**顶层 `total`**。
- **金额阈值审批**：报价/订单/合同 `approval_status` = `NONE/PENDING/APPROVED`，阈值走 `sys_config`（`approval.*.threshold`，0=不启用）。
- **字段级脱敏**：版师/打样看不到对客单价与毛利；成本/毛利限财务、管理。

## 五、建设历程（里程碑）

1. **Phase 0**：monorepo + 基础设施 + UI 设计系统（服装制造业风）+ 登录/主框架。
2. **Phase 1–3**：按《系统开发手册》修正编号规则与核心业务规则；补齐结算财务字段；打通报价→合同链路（材料合同自动带出用料快照）。
3. **8 单据逐个对齐设计稿**：基础资料 v1.3 → 样衣（双视图）→ 报价（双子表+公式）→ 订单（公式+状态机）→ 合同（抽屉+快照）→ 门户 v2.2（四步锁定）→ 对账（强校验+二级审批）→ 结算（财务口径）。
4. **产品级增强**：文件上传 / 批量导入 / 发票闭环 / 真实数据工作台。
5. **深度测试体系**：`deep-test.sh` 从 0 扩到 **493 断言**（连真 API+真 MySQL 端到端）。
6. **验收审计补口 14 项**（小+中项，4 批：结算串流程 / 合同·对账 / 合同·门户 / 付款）。
7. **生产事故止血 + 运维加固 + 一键发版**（见下）。

## 六、生产事故复盘（postmortems）

### 事故 1：发版后大面积 `Unknown column`（schema 漂移）
- **根因**：生产 `synchronize:false` + 仓库无 migration + `init.sql` 只在空库执行一次 → 本季度新增的表/列（`contract_shipment`、`company_profile`、`reconciliation_expense_item/labor_item`、`contract.revised`、`settlement.*`、`settlement_cost.tax_rate`、`factory.extra_types`、多处 `approval_*` 等）**从未落到已有生产库**。
- **止血**：幂等 `infra/scripts/hotfix-schema.sql`（存储过程守卫，只增不减）+ `upgrade-db.sh`（备份→升级→校验）。**用真 MariaDB 实测**：旧结构→升级→完整性 diff 空→二次幂等。
- **根治**：`deploy.sh` 已把结构升级内置在 API 重启之前；待引入正式 migration 体系。

### 事故 2：建补料合同必 500（组合单号超列宽）
- **根因**：`补料-母合同号-序号` 达 21 字符，超 `contract_no VARCHAR(20)` → `Data too long`。**首次对真库跑 deep-test 才抓到**（mock 单测天生看不见列宽）。
- **修复**：`contract_no` 放宽到 40；`nextWithSegment` 段限长 20（护住 `settlement_no/reconcile_no/ship_no` VARCHAR(30)）。已进 `hotfix-schema.sql`。

### 事故 3：Redis 未运行 → 不能新建单据
- **根因**：Redis 容器/镜像都没有 + 国内 ECS `docker pull` 撞 Docker Hub 超时；而单号生成依赖 Redis 且**无兜底**。
- **处置**：`health.sh` 已能区分「容器未运行/密码不一致/无响应」并兼容原生 redis；退路 `dnf install redis`；`deploy.sh` 会自动拉起**停掉的** redis 容器。**待办**：给 `NumberingService` 加 DB 兜底，拔掉硬单点。

## 七、经验教训（权威版见 `CLAUDE.md`）

- 🔴 **改 schema 必须给存量库升级路径**（改 `init.sql` 只服务全新装机）。
- 🔴 **DB/集成改动，mock 单测通过 ≠ 测过了**——必须对真库跑 `deep-test.sh`，如实汇报测了什么。
- 真机深测典型坑：组合单号列宽、分页封装顶层 `total`、DTO camelCase 白名单。
- 运维脚本坑：探活别用 `curl -f`；redis 用 `--no-auth-warning` 且分因；systemd 判定用 `systemctl cat`。
