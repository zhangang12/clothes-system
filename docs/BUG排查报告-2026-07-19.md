## 🔴 高危（8 条）

### H1. 内部创建的对账单永远无法提交复核 —— 对账主流程断死

- **位置**：`packages/api/src/modules/reconciliation/reconciliation.service.ts:123-138`（create 不占用批次）+ `:335-344`（assertBatchesStillHeld）+ `:306-315`（submit）/ `:348-358`（confirm）
- **问题**：内部端 `create` 只写 `reconciliation_shipment` 明细行，**从不占用** `contract_shipment.reconcile_id`；而后加的 `assertBatchesStillHeld` 在 submit/confirm 前检查「有明细行但无占用批次 → 抛错」，把所有内部创建的批次型对账单误判为「已退回释放的门户对账单」。
- **复现**：web 对账管理 → 新建 CONTRACT 类型对账单（UI 强制至少一条出货明细）→ 点「提交复核」→ 必抛 `该对账单为门户批次对账且已整单退回、批次已释放……`（误导性报错）。该单永远卡 DRAFT。
- **影响**：①内部对账主流程断死；②批次从未被占用 → 供应商门户可对同批次重复发起对账，同一批发货存在被两张对账单重复计费的窗口。
- **备注**：同源次要问题——create 不校验 `shipment_id` 是否存在/归属合同，`snapshot_unit_price` 纯客户端输入从不与合同快照核对（`ContractService.getSnapshotUnitPrice()` 是只为单测存在的死代码）。

### H2. web 编辑页跨单据串数据，保存即覆盖错单

- **位置**：`packages/web/src/views/layout/MainLayout.vue:56-60`（根因）+ 全部 Edit 视图
- **问题**：`<component :is="Component" />` 没有 `:key`，同一路由记录换 `:id` 时 Vue 复用组件实例，`onMounted` 不重跑，表单仍是上一张单据的内容。
- **复现**：页签栏设计即「同一编辑页不同单据是两个页签」。开「编辑样衣 #5」「编辑样衣 #7」两个页签互切：URL 已是 #7，表单仍是 #5 数据（`SampleEditView.vue:238` 的 `editId` computed 已指向 7，但 `:539` 的 `onMounted(load)` 是唯一装载入口，无 route watch）。点「保存」→ `sampleApi.update(7, 样衣5的内容)`，**把 5 覆盖写进 7**。
- **波及面**：`SampleEditView` / `QuoteEditView` / `OrderEditView` / `FactoryEditView` / `CustomerEditView`；`ContractEditView.vue:259` 更糟——`contractId` 是 setup 时取一次的 `ref`，换 id 后永远停在旧合同；`ContractEditView.vue:595-596` 从编辑页点「复制/补料」跳 `/contracts/new?copy_from=...` 同组件复用 → `copy_from` 分支不执行，保存会 update 原合同；`ReconciliationListView` / `SettlementListView` / `ExportInvoiceView` 的 `:id/view` 自动开弹框同样只在 onMounted 读参。
- **影响**：数据被静默写错单据（最高级别的业务事故）。局部曾有手动补救（`QuoteEditView.vue:492-493,564` push 后手动 `load()`），证明问题真实存在但只堵了两条路径。
- **修复方向**：`<component :is="Component" :key="$route.fullPath" />`，或各视图 `watch(() => route.params.id, load)`。

### H3. 订单尺码矩阵表尾「各 PO 合计」整行错位一列

- **位置**：`packages/web/src/views/order/OrderEditView.vue:351-360`（matrixSummary）配 `:101-126`（模板列）
- **问题**：`const fixed = readonly.value ? 3 : 4`（注释自认口径「[选择列]+款号/颜色/尺码」），但模板后来加了「洗标号/Article」列（`:106-108`），实际固定列已是 4（只读）/ 5（编辑）。`pi = i - fixed` 全部偏移 1。
- **复现**：任何打开尺码数量搭配表的订单页。单 PO 时 PO 列合计恒显示 0、真实合计串到「尺码」列下；多 PO 时每列显示后一列的合计，最后一个 PO 列恒 0。总计（TOTAL 列）碰巧是对的。
- **影响**：生产跟单按此行核对数量会被误导。

### H4. 报价列表 `?customer_id=` 参数覆盖机密行级过滤

- **位置**：`packages/api/src/modules/quote/quote.service.ts:149-153`
- **问题**：`base` 先铺 `secretCond`（`customer_id: In(可见集)`），随后 `...(customer_id !== undefined && { customer_id })` 用查询参数原值**覆盖**安全过滤。
- **复现**：无授权用户（BUSINESS/FINANCE/SUPERVISOR 均可访问 /quotes）请求 `GET /api/v1/quotes?customer_id=<不可见客户id>`。仅剩 `buyer_id IS NULL OR buyer_id IN(可见集)` 一道防线，而报价单 buyer_id 多数为 NULL → 该客户全部报价（单号/款号/金额/业务员）直接返回。
- **影响**：customer_grant 机密体系被旁路。

### H5. 客户列表「联系人检索」子串误匹配泄露未授权客户

- **位置**：`packages/api/src/modules/customer/customer.service.ts:251-253`
- **问题**：`const merged = cur ? ids.filter((x) => JSON.stringify(cur).includes(String(x))) : ids;`——`cur` 是 TypeORM `FindOperator`，序列化为 `{"_type":"in","_value":[12],...}`，用子串包含做集合交集：`In([12])` 含 `"12"`，id 为 1 的不可见客户因子串 `"1"` 命中混入结果。id 1~9 是大量 id 的子串，可利用面很广。
- **复现**：用户自建客户 id 含目标 id 子串，搜索不可见客户联系人姓名 → 不可见客户出现在列表（名称/编号/国别等）。
- **修复方向**：改为真正的 Set 交集运算。

### H6. 报价全部写操作无机密可见性校验

- **位置**：`packages/api/src/modules/quote/quote.service.ts:213`(update) / `:257`(importFromSample) / `:327`(submitQuote) / `:369`(adjust) / `:380`(toContract) / `:447`(copy) / `:479`(remove)；对照 `:173-185`(findOne) 有完整 404 防探测
- **问题**：findAll/findOne 做了行级过滤，但所有变更方法只按 id 取单后直接操作，从不调用 `visibleCustomerIds`。
- **复现**：无授权 BUSINESS 用户枚举 bigint id：`PUT /quotes/:id`（改价/改汇率）、`PATCH /quotes/:id/submit`、`PATCH /quotes/:id/to-contract`、`DELETE /quotes/:id`。
- **影响**：机密客户报价可被篡改、删除、转成订单。列表/详情 404 防探测形同虚设。

### H7. 门户「合并发货」必 400，功能实际全废

- **位置**：`packages/portal/src/views/ContractListView.vue:118-120`（前端）+ `packages/api/src/modules/portal/dto/merge-ship.dto.ts`（`MergeShipEntryDto.contract_id`）
- **问题**：`entries[].contract_id` 直接用列表返回的 `c.id`——合同主键 bigint 经 mysql2 出来是字符串；而后端 `contract_id` 只有 `@IsNumber()`、无 `@Type(() => Number)`，全局 ValidationPipe（`main.ts:16-20`）未开隐式转换 → 字符串过不了 `@IsNumber`，直接 400。
- **复现**：合同列表 →「合并发货（跨合同）」→ 勾 2 张合同填数量和物流 → 确认发货 → 必现 `contract_id must be a number` 类 400。
- **备注**：当时真栈 E2E 直接调 API 传数字所以通过，UI 真实路径从未被自动化覆盖（`__tests__`/`infra/scripts`/`packages/e2e` 全文搜不到 merge-ship）。同接口 `qty` 前端做了 `Number()` 所以没事；`createReconcile` 的 `shipment_ids` 因 DTO 有 `@Type(() => Number)` 不受影响。

### H8. 付款申请不校验对账单状态：可对 DRAFT 对账单直接付款

- **位置**：`packages/api/src/modules/payment/payment.service.ts:105-133`（createPaymentRequest 只查存在性）+ `:203-216`（addPaymentRecord 联动）/ `:331-335`（markPaid 联动）
- **问题**：对 DRAFT/PENDING 状态的对账单可建付款申请并审批付款；付清联动用的是 `update({id, status: CONFIRMED} → PAID)`，DRAFT 单匹配 0 行、静默不动。
- **复现**：业务建对账单（DRAFT，未提交/未复核）→ 财务直接建付款申请 → 审批 → 分批付清 → 付款申请 PAID，但对账单仍 DRAFT；之后再补审也只能到 CONFIRMED，**无任何路径再翻 PAID**（超付守卫已拦新申请）。
- **影响**：①对账二级审批（含超发闸门、发票=对账金额校验）被付款侧整体架空；②结算聚合只认 CONFIRMED/PAID：已付的钱完全不计入总货款，利润算错。
